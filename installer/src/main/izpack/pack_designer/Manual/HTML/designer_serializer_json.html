JSON writer


Topic content

Description

The JSON writer is a serializer that transforms a structured Message (internally represented as DOM tree) into an JSON data stream.

Creation

A JSON writer is created, like all other serializers, by selecting it in the according Stream channel.

Configuration

The user can configure the JSON writer using the following dialog.

 

The attribute Readable output can be set in any mode. It inserts newlines and tabs into the output to make it readable for humans.

Actually the serializer can work in three different modes:

Generic mode
Writing repeated elements as JSON array

That means that no schema and no message type is configured. All information needed to write the JSON stream must be contained in the message.

Especially information about arrays has to be contained in the XML document. E.g. if you have the following XML document:

<json>

 <ident>test1</ident>

 <data><key>kg</key><value>Kilogramm</value></data>

 <data><key>m</key><value>Meter</value></data>

</json>

The JSON writer will create the following (wrong) JSON document:
 {
  "ident":"test1",
  "data":{"key":"kg","value":"Kilogramm"},
  "data":{"key":"m","value":"Meter"}

 }

This is wrong, because the field "data" exists twice.

To work correctly the repeated elements need to be marked as array elements:

<json>

 <ident>test1</ident>

 <data isArrayElement="true"><key>kg</key><value>Kilogramm</value></data>

 <data isArrayElement="true"><key>m</key><value>Meter</value></data>

</json>

In this case the JSON writer will create the following correct JSON text:

{

  "ident":"test1",
  "data":[{"key":"kg","value":"Kilogramm"},{"key":"m","value":"Meter"}]
}

There is one possibility to avoid the need to mark array elements: If the root element contains a sequence of records and there are no other repeated elements in the XML document, the user may set the check box named Message contains array. In this case the following XML message

<Records>

   <Record>

       <ident>001023</ident>

       <name>SevenUp</name>

       <amount>100</amount>

       <price>2.89</price>

   </Record>

   <Record>

       <ident>001025</ident>

       <name>Bluna</name>

       <amount>120</amount>

       <price>1.49</price>

   </Record>

</Records>

is written as the following JSON text:
[

 {

         "ident": "001023",

         "name": "SevenUp",

         "amount": "100",

         "price": "2.89"

 },

 {

         "ident": "001025",

         "name": "Bluna",

         "amount": "120",

         "price": "1.49"

 }

]

Writing simple data values

In generic mode all simple data values are written as JSON string, e.g. "test" or "0012".

If you set the check box Consider xsi:type XML attributes named "xsi:type" are considered, where xsi is a prefix for the namespace http://www.w3.org/2001/XMLSchema-instance.

Given the following simple XML document

<Data xmlns:xsd="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">

 <name>Test</name>

 <amount xsi:type="xsd:int">33</amount>

 <size xsi:type="xsd:decimal">10.2</size>

</Data>

 

the JSON serializer would write:

{

 "name": "Test",

 "amount": 33,

 "size": 10.2

}

 

If Consider xsi:type is not set it would write:

{

 "name": "Test",

 "amount": "33",

 "size": "10.2"

}

Luckily using a graphical mapping you can set the xsi:type attributes automatically, by simly setting the option write schema instance type in Orchestra mapping, section general settings.

Because in generic mode there is not information about the cardinality of elements, all complex elements are written as objects.

Schema mode

The Information contained in the selected XML schema is used to serialize the message to JSON. Typically this is used together with FHIR schemas, because FHIR defines a specific mapping from XML to JSON.
So the FHIR message

<Bundle xmlns="http://hl7.org/fhir">

   <type value="searchset"/>

   <entry>

       <resource>

           <Patient>

               <name>

                   <family value="Link"/>

                   <given value="Link"/>

               </name>

               <gender value="m"/>

               <birthDate value="1963-06-10"/>

           </Patient>

       </resource>

   </entry>

</Bundle>

 

is written as the following JSON document

{

"resourceType": "Bundle",

"type": "searchset",

"entry": [

 {

  "resource": {

   "resourceType": "Patient",

   "name": [

    {

     "family": [      

      "Link"

     ],

     "given": [      

      "Link"

     ]

    }

   ],

   "gender": "male",

   "birthDate": "1963-06-10"

  }

 }

]

}

 

If you have another, non FHIR schema, the serializer uses the information in the schema to write arrays if the cardinality of the element is > 1 and to write JSON values according to the type if the simple elements.

Note that you don't need to mark repeated elements using XML attributes like in generic mode.

Typed mode

The information contained in the selected message type is used to serialize the message to JSON. The serializer uses the information in the message type to write arrays if the cardinality of the element is > 1 and to write JSON values according to the type of the simple elements.

The following XML document

<json:Records xmlns:json="http://emds.de/records" xmlns:types="http://emds.de/types">

 <json:Record>

   <json:ident>0001002</json:ident>

   <json:size>501</json:size>

   <json:weight>3.14</json:weight>

 </json:Record>

 <json:Record>

   <json:ident>0001003</json:ident>

   <json:weight>2.12</json:weight>

   <json:property><types:name>created</types:name><types:value>2016-11-13</types:value></json:property>

   <json:property><types:name>author</types:name><types:value>Thomas</types:value></json:property>

 </json:Record>

 <json:Record>

   <json:ident>0001004</json:ident>

   <json:size>301</json:size>

   <json:weight>1.5</json:weight>

   <json:names>Hans</json:names><json:names>Georg</json:names>

 </json:Record>

</json:Records>

 

will the be written as the following JSON document

[

 {

         "ident": "0001002",

         "size": 501,

         "weight": 3.14

 },

 {

         "ident": "0001003",

         "weight": 2.12,

         "property": [

                 {

                         "name": "created",

                         "value": "2016-11-13"

                 },

                 {

                         "name": "author",

                         "value": "Thomas"

                 }

         ]

 },

 {

         "ident": "0001004",

         "size": 301,

         "weight": 1.5,

         "given name": [

                 "Hans",

                 "Georg"

         ]

 }

]

Note that you don't need to mark repeated elements using XML attributes like in generic mode.

Note also that within the context of the REST services you cannot select a specific message type. Instead you select the entry <configured type>.

JSON writer configured in a REST service

In this case the message type is used which is configured in the request or response of the REST service using the Serializer to write JSON data.
