Implementing a REST web service


Topic content

Description

Using Orchestra, you will be able to implement REST web service of your own, without a need for a specific programming language knowledge. In this particular case, Orchestra app will act as an external REST web service which exposes different APIs based upon your requirements. The Rest Service Provider concept used for creating web service contains only the declaration or REST methods that the user wants to implement. The actual implementation is done by creating process model object and assigning it to particular REST method as its implementation.

Problem to Solve

Imagine that you need to implement some basic REST web server which will expose different APIs in order to perform various CRUD (create, read, update and delete) operations against some database Resources. In the following example, we will try to demonstrate you this concept by using simple domain model describing a Person.  We will implement several different REST methods which will allow you to perform actions like, creating,  listing and deleting particular person entries. The Person entries will be stored in simple MySQL database containing single database table with following information: id, name, surname, gender. In the following example, we will use Mozilla Firefox browser as a REST client and a simple MySQL DB as a database storage. The following diagram illustrates the example.

Implementing REST Service Client - example diagram

Under the hood, Orchestra provides you with the mechanisms for mapping different structures and message formats. In our example, you will see how the person data is transformed from one representation state to another using XML, JSON and DB table data formats.

Step Overview

In order to successfully create REST web service  using the Orchestra, several steps need to be followed.

1.Create Orchestra REST Service Provider by importing WADL description file

2.Configure previously created REST Service Provider object

3.Create a process models as an implementation for REST Methods

4.Configure POST REST web service method for creating new person

5.Configure DELETE REST web service method for deleting a person

6.Configure GET REST web service method for retrieving person list

Prerequisites

You will need to create a simple MySQL database containing a Person table. Enable access to this database for a particular user which will be used in order to make a successful database connection. Some basic knowledge for creating Orchestra process models and different database channels is required.

Used Concept

Orchestra introduces REST Service Provider  concept which can be used in order to implement REST web service.

Steps

Step 1 - Create Orchestra REST Service Provider by importing WADL description file

In our example, we will create REST Service Provider by importing WADL description file. You can also create REST Service provider manually. Here is the example of manual creation for REST Service provider used in this topic. For more information about REST Service Provider please refer to REST Service Provider reference manual.

The following code represents the content of the WADL file used in the example described in this best practice story.

Example - WADL file

<?xml version="1.0" encoding="UTF-8"?>

 <application xmlns="http://wadl.dev.java.net/2009/02" xmlns:xsd="http://www.w3.org/2001/XMLSchema">

   <resources base="/resttest">

     <resource path="/Person/{persid}">

       <doc title="References a certain person identified by its person-ID"></doc>

       <param name="persid" style="template" type="xsd:string" required="true"></param>

      <method name="POST">

         <doc title="Add a person with given ID"></doc>

         <request>

           <param name="Content-Length" style="header" type="xsd:int" required="true"></param>

           <param name="Content-Type" style="header" type="xsd:string" required="true"></param>

           <representation mediaType="text/xml" element="Person"></representation>

         </request>

         <response>

           <param name="Content-Type" style="header" type="xsd:string"></param>

           <representation mediaType="text/xml" element="Person"></representation>

         </response>

       </method>

      <method name="DELETE">

         <doc title="delete Person"></doc>

         <request>

           <doc title="parameter persid"></doc>

         </request>

       </method>

     </resource>

     <resource path="/personlist{?name}{?surname}">

       <param name="name" style="query" type="xsd:string" required="true"></param>

       <param name="surname" style="query" type="xsd:string" required="true"></param>

      <method name="GET">

         <doc title="Retrieve persons list by name and surname values"></doc>

         <request></request>

         <response>

           <param name="Content-Type" style="header" type="xsd:string"></param>

           <representation mediaType="application/json"></representation>

         </response>

       </method>

     </resource>

   </resources>

 </application>

 

After importing the WADL file the Rest Service provider object will be created, containing the highlighted methods from the example above.

Step 2 - Configure previously created REST Service Provider object

This step will describe details of previously imported REST service provider. We will also apply some additional changes in order to set up our REST service properly.

•In opened configuration panel of previously created service provider select Resources element.

The base path was automatically generated in order to provide unique value.

•Check overwrite default path option and enter custom base path value.

The base path value is a part of URL-Pattern of the web service calls and, therefore, we change it to more user-friendly value. See image below.

Implementing REST Service Client - resources configuration

•Select the Resource element of Rest Service provider.

In the configuration panel, you will see the additional settings for the request URI. The path value contains the {persid} parameter which will be used for each method implemented by this web service. We will use this parameter as a unique identifier for the person object inside our MySQL database and based upon it, we will perform create, delete and list actions against the database. See image below.

Implementing REST Service Client - resource configuration

•By clicking on the Method object of the service provider, the additional setting for each method will be displayed in the configuration panel.

The Name field is actually the type of REST method. Here you can also see the URL-pattern of selected message API call as well as the Response media type of the message.

 

Implementing REST Service Client - method configuration

The most important part of this configuration is Implemented by value. Here is the actual implementation of your REST method. The implementation itself needs to be defined in the process model and this process model needs to be assign to this REST method. This is done by "technical mapping process" which can be invoked by clicking on the Change technical mapping button. In The Steps 4, 5 and 6 we will go into detail implementation for each process model as well as through the process of technical mapping for each of three supported REST method.  

 

Step 3 - Create a process models as an implementation for  REST Methods

This step will describe details about the implementation of a POST REST web service call for creating a new person. Please note that for each REST method declared in your REST Service provider you need to create a corresponding process model as an implementation of the method. At the end of this step, we will show you a complete list of process models and channels used for our example without going into details about its implementations. All three process models are very similar with the differences in Database Outbound channels used in them for accessing database layer.

The process model named pm_PersonAdd used for this implementation is using Database Outbound Channel for accessing database resources for a person domain model. See image below.

Implementing REST Service Client - pm_PersonAdd process model

 

The input parameters of this process model are person id (NEWPERSONID)  value and an XML message (NEWPERSON) containing information about new person entry we want to create. See image below.

Implementing REST Service Client - pm_PersonAdd process model properties

The Database Outbound Channel ch_db_out_create_person used in this process model will create a new person entry. The input parameters for the channel are id, name, surname and a gender information about a person. The output message of the channel call will contain information about created person entry. See image below.

Implementing REST Service Client - ch_db_out_create_person channel

Note: In order to connect to desired database (in our case DB containing person entries) using Database Outbound channel, you will need to create database connection environment entry (ee_person_db). In the image below you can see the environment entry configuration for accessing MySQL "restexample" database used in our example.

Implementing REST Service Client - database environment entry

The input parameters of the channel are extracted from the NEWPERSON using xPath expressions. This is configured in the call activity of the process model. See image below.

Implementing REST Service Client - call activity configuration

The same way, the output message of the channel will be forwarded to the REST Service provider and will be shown as the response of the GET method call. See image below.

Implementing REST Service Client - call activity configuration

To summarize the process model implementation. The process model receives information about a person, contained in the XML message together with the person id. It extracts the values from the message (NEWPERSON) and performs a call to a channel with the extracted input parameters. The output message (person) of the channel execution will contain information about created person and it will be assigned as an output value of the process model.  The next step is to integrate the process model as an implementation of the POST method in our REST service provider.

Step 4 - Configure POST REST web service method for creating new person

This step will describe details about the implementation of a POST REST web service call for creating a new person entry.

•Open previously created REST Service Provider.

•Click on the Resources > Resource: /Person/{persid} > Method POST to open the configuration panel for the method.

The selected method has one request parameter called persid. This parameter is configured in the Resource parent element of this method.

The request and response are in XML format. You can change those values by opening Request and Response elements of the selected methods.

Implementing REST Service Client - POST method configuration

•Click on the Change technical mapping button to open Configure method implementation dialog.

Within this dialog, you will find three different views for configuring the implementation of your REST method. Technical mapping, Request mapping, and Response mapping views are available.

•In the Technical mapping view, select the process model for this method implementation.

Here we are going to use the pm_PersonAdd process model created in step 3. See image below.

Implementing REST Service Client - POST method technical mapping

•In the Request mapping view, assign persid and MSG (the message containing the body of the POST request) values to corresponding input parameters of the selected process model. See image below.

Implementing REST Service Client - POST method request mapping

This mapping allows you to forward request data into the process model. Based upon process model implementation, described in step 3, new person object will be created and the result data contained in the NEWPERSON message will be returned  back.

•In the Response mapping view assign NEWPERSON message (output message of the process model) to the response of the method call. This message will be shown as the response of the POST request call. See image below.

Implementing REST Service Client - POST method response mapping

•Save the changes and execute POST request in order to test the method implementation. You can find URL-Pattern of the method call in the configuration dialog.  See image below.

Step 6 - Configure GET REST web service method for retrieving persons list

This step will describe details about the implementation of a GET REST web service call for fetching a person lists by person name and surname values.

•Open previously created REST Service Provider.

Implementing REST Service Client - GET method resource configuration

As you can see in the image above there are two different Resource elements. The Resource: /Person/{persid} element contains POST and DELETE methods described in steps 4 and 5 accordingly. We have introduced the new Resource element, Resource: /personlist{?name}{?surname}, for the third method because the URL-pattern is a little bit different. For the first two methods, we used the same URL pattern containing the person id (persid) request parameter. For the GET method described in this step, we defined two request parameters, person name and person surname, which will be used in order to retrieve the list of persons. For this method  instead of Person, we also have different URI value personlist, which is better describing the actual REST method request.

•Click on the Resources > Resource: /personlist{?name}{?surname} > Method GET to open the configuration panel for the method.

Implementing REST Service Client - GET method configuration

Summary

In this topic, we introduced you with a concept of REST Service provider and the mechanisms to leverage different Orchestra concepts in order to create your own REST web service.  You saw how to easily create REST Service provider object by importing the wadl description file.  We showed you have to integrate a process model as an implementation of your REST method by using technical mapping functionality described in previous steps.  By using Database Outbound channels inside the process models Orchestra gives you a possibility to access different database resources and perform CRUD operations by executing REST web service calls against implemented REST web service.
