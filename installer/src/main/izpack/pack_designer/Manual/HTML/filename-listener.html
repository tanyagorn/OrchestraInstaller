Filename Listener


Topic content

Description

The Filename Listener is an Inbound channel which regularly searches for new files in a particular directory and returns  meta data about the recognized files, specifically the name of the file. Whenever a file in the observed directory is found whose filename matches the search pattern a  start event is created containing the name of the file.

Actually the created meta data is a structured message having the following form:

   <Standard.FileInfo>

       <FILENAME>

       <FILENAMEFULL>

 

The message element FILENAME contains the original name of the file (it may be renamed by the channel after recognizing it). The message element FILENAMEFULL contains the full filepath of the original file.

The values of FILENAME and FILENAMFULL are also delivered as properties of the returned Message.

The original filename is also returned from the channel as value of the output parameter origFilename. If the Channel is configured to rename the file, it has an additional output parameter named changedFilePath returning the full path to the file after renaming it. The Process model may use this value to read the file using the File Reader channel.

This channel basically is an alternative to the File Listener channel. But in cases where the content of a file  is not always readable, specifically when using network file systems, it might be reasonable to use the Filename Listener instead. It recognizes new file names in the directory and then creates an instance of the connected process model. Then the process instance is responsible for reading the content of the file. The process developer may then react appropriately if the file is not readable, e.g. try it several times or finish with an appropriate error.

Another reasonable use case for this channel is if the existence of an input file serves as a trigger to start a new process model and the content of that trigger file is not relevant.

Creation

To create a Filename Listener follow the steps described in the general description of Channels

Configuration

The channel is configured using the following panel

Configuration of the filename listener channel

•Directory Most important the user has to select the directory which is observed for new files

•Authentication If the directory is a network directory probably Orchestra has to authenticate itself. In this case the user has to select an appropriate User credential

•Name pattern Normally only specific files are to be searched. Therefore the user has to declare a filename pattern here.

•Polling period

Typically the channel searches its source for available data periodically:

To change the intervals at which the channel searches its data, the user clicks on the button labeled with three dots and then can edit the timer properties.

There are two types of timers, periodical timers and absolute timers. Both are configured using the same dialog. In most cases a periodically timer is used which regularly creates a signal.

timer configuration

•Description for the timer, e.g. polling timer

•Valid from/to Here the user can state a time span where the timer is valid. Outside of this time span, the timer is invalid and doesn't create events.

•Mode This Option is only sensible if the timer dialog is used to configure a timer event. In the context of channels it is disabled.

•Trigger type The user may restrict the timer to work only at certain days:

oDaily Here the user may restrict the timer to fire events on certain days of the week

oMonthly Here the user may restrict the timer to fire events at a particular day of the month

•Absolute If an absolute timer is configured, the user must enter a fixed time at which the timer raises an event. E.g. one could declare the timer to run every day at 6:00 in the morning.

•Periodically If an Periodical timer is used the user has to configure the following properties:

oStart/End time A time span of the day where the timer shall be active

oRepeat count Here the user may restrict the timer to fire only a certain amount of events after start. E.g. one could declare the timer to start at 6:00 AM an fire exactly 2 times. The standard value is infinite, that means there is no restriction.

oRepeat interval the time interval after which an event is fired periodically.

 

•Work suffix whenever the channel recognizes a file it is renamed by adding the work suffix. In the case of the Filename listener this is hardly ever used.

•Delivery

Here the user may change the order in which the recognized files are read. This is rarely used but may be necessary if the associated process model works in FiFo mode and the files must be processed in a defined order.

•Post processing

After the file name was read  it should be renamed, deleted or moved to another place to ensure that it is not processed a second time.

The user may select an action to be performed when the file could be processed sucessfully (Success action) and another action to be performed if in the course of file processing an error occurred (Failure action)

In each case there are four possibilities:

•Rename the file by adding a certain extension to its name

•Move the file to another directory

•Delete the file

•Move & rename the file

For the Failure action the user may also decide to to nothing (No action)

Caveat: This causes the file to be processed infinitely as long as it exists!

 

Testing the channel

You may test the Filename listener like every other inbound channel.

When the channel is configured the user can test its functionality by clicking on the test symbol in the tool bar of the Designer.

Then a dialog opens showing the events the channel receives or an error message.

 

See also

Other channels to read files are the Inbound channel File Listener, and the Outbound channels File Reader and Multi File Reader. To write files you normally use the File Writer.
