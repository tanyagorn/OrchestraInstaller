File archive


Topic content

Description

This filter is available as input as well as output filter.

Both, the input as well as the output filter, write a copy of the handled byte stream into a file for archiving purposes. This feature can be used to write e.g. HL7 messages in its original form to a file.

Creation

See the description in Stream filters

Configuration

To configure the filter double click on its name. A dialog appears where you can configure the filter:

Properties of the file archive stream filter

In the example configuration above you see that in certain fields like Subdirectory or Filename you can enter patterns. Within these patterns parameters are defined using the form $(parameter). When these patterns are evaluated at runtime each parameter is replaced by a string value.
The following parameters are supported:

•$(ctrl:counter) replaced by a unique identifier

•$(dt:<date/time format>)  Replaced by a date/time string. the date/time format is defined by the simpledateformat of java.

•$(msg:ID) replaced by the message identifier of the current Orchestra message

•$(msg:propertyName) replaced by a message property contained in the current message. Most useable in this context is probably the property named FILENAME

•if none of the above pattern matches, the parameter is treated as a simplified xpath expression. E.g. $(tables/EDI_DC40/docnum)

Step by step

•Name In this field the user may change the name of the filter step to a descriptive name

•Description Here the user may enter an additional description

•Operation Mode There are two operation modes: Create/Overwrite and Append if exists. The first one overwrites existing files, the second one appends content to existing files. If you can ensure unique file names (by using appropriate file name patterns) the Create/Overwrite mode is appropriate. Otherwise the Append mode might be the better choice.

•Cache There are two caching modes: Write through and Write on close. The first one writes each byte that is written to the output channel to the archive file without delay, the second one keeps the message in memory until the whole message was written, and writes it to the archive afterwards. With very big messages the Write through mode is advisable, if the target system refuses the message if it is not complete, the Write on close mode might be appropriate.

•Base directory This environment entry defines the base directory where the archive files are to be written to.

•Authentication if the base directory is a network directory probably Orchestra has to authenticate itself. In this case the user has to select an appropriate User credential

•Subdirectory Optionally you can enter here the name of a subdirectory where the archives are to be written to. The subdirectory may contain patterns as described above to structure divide the archive files in partitions. An example is given above: arch_$(dt:yyyy)/$(dt:MM) means that for each year a directory arch_<year> is created and below this for each month a subdirectory named like the number of this month is created.

•Filename Here the name of the output file must be entered. The file name may contain patterns as described above. E.g. An example is given above: out_$(dt:dd).txt means that file names are created containing the day of the month like out_30.txt. Other sensible file name patterns might be msg_$(dt:HHmmss)_$(msg:FILENAME).txt or rfc_$(tables/EDI_DC40/docnum).txt

•Encoding The encoding for the text file is selected in this box.

•Header In this field the user may enter a text to start a message within the output file. This is most helpful when working in Append mode. Above you see a sensible pattern one can use in this header:
-- Received at $(dt:HH:mm:ss), File $(msg:FILENAME), Msg $(msg:ID) ---- \r\n
Using such patterns the an archive file is probably better searchable.

•Trailer In this field the user may enter a text to terminate a message within the output file. This propably is most helpful when working in Append mode.
