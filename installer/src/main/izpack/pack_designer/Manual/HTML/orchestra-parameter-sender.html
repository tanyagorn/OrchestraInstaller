Orchestra Parameter Sender


Topic content

Description

Orchestra Parameter Sender is used to create a connection between two Orchestra nodes. It is build up like signals, but with a point-to-point connection.   It is used invoke functions between distributed orchestra instances.

 

The inbound channel implements a "idempotent" behavior. This means, the receiver can determine if the current request is sent repeatedly. When it detects such a repetition the request is skipped and not executed anymore. The sender receives an "already-executed" result. The sender can define if idempotent communication should be used or not. Per default, the idempotent behavior is active. When the parameter "IDEMPOTENT" is set to false, this behavior is explicitly disabled. This parameter controls the behavior of the channel. Normally the channel implements a "idempotent" behavior as default. This means when the same call is invoked twice, it will only be executed one time. This is reasonable, since orchestra ensures the execution of the remote started process. Sometimes this behavior may not be reasonable. Therefore it can be disabled with setting "IDEMPOTENT" to false. In this case every retry starts a new invocation of the remote function.

Configuration

The dialog to configure the Orchestra Parameter Sender looks like:

base view of the channel orchestra parameter sender

Step by step

•Endpoint Here you may select Static endpoint or Dynamic endpoint. If Dynamic endpoint is selected, the channel has a parameter ctrl_endpoint. The caller has to provide the actual URL as value of this parameter. In this case the URL defined in the Service-URL is not used; If a Proxy server is defined in the Service-URL this proxy is used even if Dynamic endpoint is selected.

•Service-URL - Choose an Environment Entry of type Uniform resource locator  which points to the remote Orchestra. It contains the URL to call if Static endpoint is selected. If in the Environment entry a Proxy server is defined it is used even if Dynamic endpoint is selected.

•Read-Timeout - If after this time the receiver returns no data (in synchronous mode) an error occurs.

•Authentication - HTTP authentication type: Basic: User/Password, Digest: User/Password , Certificate authentication: authentication of the clients with a certificate

•User-Credentials - (Optional) Choose a user

•Mode - Defines the mode of operation. When set to request, orchestra submits the request, but doesn't wait for completion. Otherwise, when synchronous is used, the execution blocks until a result is received

•Parameters - Define one or more parameters to be passed to the remote Orchestra

•the mapping of the parameters between outbound and inbound channel is done by name (order of the parameters is not important)

•the parameters must be set manually at outbound and inbound side (there is no export/import)

•the usage setting (INPUT, OUTPUT) is only used in request/response mode.

Testing

In the Scenario with the Parameter Receiver channel start the test engine, open the tab Channels, open the Properties of the Parameter Receiver and from there copy the URL. With this URL you man then configure the Service-URL.
