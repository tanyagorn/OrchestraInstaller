Debugging Java code


Topic content

Description

In Orchestra, it is possible to develop Java code, for example by writing a Java source file or a Java extended mapping. If it isn't working as expected, the developer would like to debug his code and follow step by step what is happening. The developer also sometimes doesn't know exactly what the parameters of some methods mean.

Problem to Solve

Orchestra itself has some features like code completion and coloring, but does not provide a debugger. It also doesn't tell the names of the method parameters but only the data type. So now here follows an explanation to achieve that.

Prerequisites

You need a running Orchestra installation and you need to install eclipse and Java.

Used Concept

The concept is to import the Orchestra in eclipse and develop the classes here. That will be familiar to all Java developers that use eclipse. You can then reference the classes you developed in eclipse from the designer and debug them with the eclipse debugger. After checking that everything works fine the content of the classes can be copied to the Java file in the designer and it is sure that it's now working.

Steps

Creating an eclipse project

Open eclipse and select File->New->Java Project. Give it a name (like OrchestraCustomerDesignerDebugging in this example) and click on "Finish".

Importing Orchestra binaries and libraries

Now you have to tell eclipse where it finds the binaries and libraries that are necessary to launch Orchestra. First, we add the Orchestra classes. Right click on the project folder and select Properties. Then click on "Java Build Path" in the menu on the left. If you now select "Add External Class Folder..." a new window opens where you can select the folder to the Orchestra classes. In our case, the folder has to be <tomcat>/webapps/Orchestra/WEB-INF/classes/.

Adding orchestra classes to the build path

 

Now we want to add the Orchestra libraries. First, click on the "Add Library..."-Button. Then select "User Library" and click on "Next".

Adding a new library

 

Now you can see a window with all your configured user libraries. For adding the Orchestra libraries, we need a new one. Click on the button "User Libraries.." and select "New..." in the next window.

Creating a new user library

 

Now enter a name for the library. We use one library called "Orchestra" and one called "Orchestra_designer". Now select the "Orchestra"-library and click on "Add External Jars...". Navigate to the folder <tomcat>/webapps/Orchestra/WEB-INF/lib/ and select everything that is inside. Then click on "Open". Now you can see all the jars in the library. Do the same with the "Orchestra_designer" library and the folder <tomcat>/webapps/Orchestra/lib_designer/.

Adding jars to the user library

If you have finished, click on the "OK"-Button. Now you can still see the window with the two configured libraries. Tick both in the list and then select "Finish". Now all the resources needed for running Orchestra from eclipse are available.

Developing classes in eclipse

Now you can start with developing your classes in eclipse. Just add a new package (like de.soffico.mapping) and create new classes (like CallFromMapping). From this classes you can reference all libraries and classes provided by Orchestra. Just import them. Eclipse will also show you the name of the method parameters and not only the data types. That can give you more information about how to use the methods.

Starting Orchestra designer from eclipse

If you have finished developing your classes (or even before, if you want to test if the designer is running), you need to configure the debugging settings to launch the Orchestra designer. Select Run->Debug Configurations... from the menu.

Navigating to debugging configurations

 

Now a new window opens and shows all possible configurations. At the menu on the left, double click on "Java Application" to create a new configuration. Give the configuration a name (in the example it is called "OrcDesigner") and make sure that your Orchestra designer project is selected. Now you need to enter the main class, which is emds.epi.graph.EPIEditor in our case. Then click on "Apply" and select the "Arguments" page.

Configuring the debugging settings

 

Here you need to enter the following text in the VM arguments textfield: -showversion -Xmx256M. With this arguments the designer runs properly. Now click one more tome on "Apply" and then select "Run" to launch the designer.

Configuring the debugging arguments

Calling developed classes from Orchestra scripts or mappings

Now you can access your code from the designer. If you create an extended java mapping or a java source file, you can import the classes as usual by defining the package where they are located.

Referencing a developed class from the designer

Break point usage for debugging

If you now add a break point anywhere in the developed class (like in the example above in the line with "return "hallo") and then test the scenario element as usual, you will find a window saying "please wait" opening in the designer when the break point is reached. You can debug how you want now. If you are sure that everything works, you can copy and paste your code back to the scenario element and use it again without eclipse.

Summary

In order to debug Java code in the Orchestra designer, it is necessary to import the class files and libraries to a new eclipse project. In this project, a developer can add classes as he wants and start the designer from the class emds.epi.graph.EPIEditor. In this designer, the scenario elements can reference the developed classes as in usual Java code. By testing the scenario elements inside the designer, the designer will stop at any break point and the developer can debug inside eclipse. In the end, the code needs to be copied back to the scenario element in order to not depend on eclipse any more.
