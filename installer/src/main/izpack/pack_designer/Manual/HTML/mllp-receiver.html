MLLP Receiver


Topic content

Description

MLLP is a shortcut for Mini Lower Level Protocol.

Purpose of object

MLLP is used to exchange HL7 messages. This channel is used for receiving HL7 messages from a network.

Creation

To create a MLLP Receiver follow the steps described in the general description of Channels

Configuration

The dialog to configure the MLLP Receiver looks like:

base view of the channel MLLP receiver

Step by Step

1.Choose a Mode in general settings.

 - Asynchronous: a process is only invoked

 - Synchronous: a process is invoked, it will be waited untill it finished

2. Choose a MLLP Version

 - MLLP version 1: version without acknowledgement on network protocol level

 - MLLP version 2: version with acknowledgement, this acknowledgement will be send on network protocol level

3. Frame characters: Here it is possible to define the start and the end of your messages.It can be defined for in- and outbound messages.

4. Connection test

 - Raise event on client connect or error: Orchestra reports every change in the connection state as system event

 - Raise event on error: Orchestra reports only errors as system event

5. Choose TCP server or client. Then an appropriate  environment entry must be chosen.  In almost all cases the MLLP Receiver acts as TCP server.

6. Synchronous timeout: Choose the maximal duration within Orchestra must answer a message. If the process model doesn't send a message in this duration a error message will be send as HL7-Acknowledgment.

7. Message handling:

•For inbound processing the HL7 Message Reader should be chosen, also it is possible to choose a message type as generic type here.

There are two parsing modes parse on creation (the default) and parse on demand.

•parse on creation - the data are parsed by the selected deserializers while reading the byte stream. If in this process an error occurs, it is visible only in the log file because no process instance exists yet.

•parse on demand - the byte stream is read binary without interpreting it; together with the data the configuration of the selected deserializer is stored. If the message is accessed, e.g. by a mapping, it is parsed using the attached deserializer configuration. If the deserializer throws an error it can be handled by from the process instance.

•For outbound processing in synchronous mode, the HL7 Message Writer should be chosen. Here it is also possible to choose a message type.

8. HL7 Acknowledgement:

 -enabled:

         - in asynchronous mode: a standard answer will be send, the only errors that can be considered are errors in the channel

         - in synchronous mode: Orchestra will wait till the process model has finished. If the process model sends a HL7 answer, nothing is done,

           if the process model didn't send an answer, the answer is send from the channel, error messages from the process can be considered here.

 -disabled:

         - in asynchronous mode: no message is send

         - in synchronous mode: The process model must send an answer, if it doesn't, no message is send.

 - encoding: choose an appropriate encoding

 - messages: for more information how to change the messages click here

 - Ack serialization: here can be specified what can be done in case of an error
