Working with a shared scenario


Topic content

When a scenario has been shared, a user can create and modify scenario elements as expected. The versioning state of scenario elements are displayed within the scenario element tree. When versioning is active, symbols are added that show the versioning state of the current scenario element. In the following, a scenario is shown that contains two newly created elements (pm_test_process_modell,ch_java_channel) and one element that is already stored in the versioning system but was modified locally.

 

States of a versioned scenario elements

 
Available scenario states:

? - The scenario element was created but was not stored at the versioning control system

* - The scenario element is stored at the versioning control system, but the local copy was modified

< - Ingoing change from the versioning system

 

Storing changes to the versioning control systems

When the changes of the scenario element should be available for other users, the changes must be pushed to the versioning control system. The required function is "Team synchronization" and can be found in the context menu of the scenario tree.        

Invoke the team synchronization

 

The team synchronization dialog looks as follows:

Team synchronization dialog

The above dialog is the main dialog when changes from or to the versioning system has to be submitted. When scenario elements are stored to the versioning control system one have to give a short and detailed description. Both fields are useful when the changes of the scenario should be traced.

 

In the dialog above the context menu is shown that provides the basic functions that can be used.

 

•Refresh remote changes: Check the current state of the scenario within the versioning control system. If newer or changed scenario elements are detected, a hint is given in the tree of scenario elements

•Check in remote changes (without conflicts): Push all of the changed scenario elements to the versioning control system. This operation can only be applied when the scenario elements are not in conflict. This means nobody else has changed the scenario elements meanwhile. Note: When a scenario element is stored that references other elements this elements will be stored as well (only for newly created elements). This is necessary since orchestra ensures always a consistent state of the scenario within the versioning control system.

•Check out remote changes (without conflicts): Pull all changed scenario elements from the versioning control system. This is only possible when there is no conflict

•Resolve conflicts with check-in (Force push): When a conflict occurs, it can be resolved with this function. The local changes are pushed to the versioning control system and the remote changes are overwritten.

•Resolve conflicts with check-out(Force pull): When a conflict occurs, it can be resolved with this function. The local changes are overwritten with the version from the versioning control system

•Revert local changes (without conflict): The local changes are overwritten with the state from the version control system.
