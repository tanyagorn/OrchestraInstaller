Mapping functions


Topic content

 

In the Orchestra mapping as well as in the Extended Java mapping you can use the functions defined in the following sub chapters.

In the graphical Orchestra mapping you select the wanted function from the panel named Mapping; see attribute mapping using predefine functions for details.

In the procedural mapping (aka Extended Java mapping) you simply use the functions in the mapping script. Actually the functions are implemented as static methods of the class InternalMappingFunctions. For each mapping (procedural as well as graphical) a Java class is generated which inherits from this base class.

List of available function groups

•Conversion functions used to convert values of different data types

•Date functions functions with values of class XMLGregorianCalendar

•Logical functions handling boolean values. Currently only not(boolean) is defined here

•String functions working with string values; some of these functions may have different names in the graphical mapping and in the procedural mapping. If the function is defined in XPath, in the graphical mapping that name is used. If this name then is not allowed in java due to syntactical reasons another name is used there. E.g. the XPath function starts-with is named stringStartsWith in java.

•Context functions the two functions position and getFilterPos

•IDOC functions normally not used when creating a mapping manually. The functions defined here are used from the IDOC mapping.

•DICOM functions the two functions getOrchestraInstanceUID and buildPN. Normally only used in Mappings dealing with DICOM message types. This is typically necessary if the DICOM worklist server is used.
