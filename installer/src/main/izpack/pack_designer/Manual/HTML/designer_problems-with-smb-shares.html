Logging jCifs

DFS deactivation (if not used)

# jcifs.smb.client.dfs.disabled = false

change to

# jcifs.smb.client.dfs.disabled = true

 

It is possible that jCifs believes that it finds a DFS-System and tries to reach the computer and always receives timeouts. If you know that Orchestra does not communicate with a DFS globally, you should deactivate this parameter and therefore deactivate DFS.

For that reason, one possible source of errors within the SMB communication is excluded.

SMB Logging

The parameter  jcifs.util.loglevel in the file jcifs.properties should be set to the value "3".

You can find the configuration file here: <tomcat>/webapps/orchestra/WEB-INF/classes/config/jcifs.properties

 
------------------------------------
# JCIFS properties for Orchestra
 
# See http://jcifs.samba.org/src/docs/api/overview-summary.html#scp
# for a detailed description of all properties
 
# An integer specifying the verbosity of log messages. Higher values are more verbose
#   0 - No log messages are printed -- not even crticial exceptions.
#   1 - The default level. Only critical messages are logged.
#   2 - Hightened log messages suitable for logging while under load.
#   3 - Almost everything.
#  >3 - Debugging only.
jcifs.util.loglevel = 3                                    
------------------------------------

 

The jcifs library generates extended output via System.err.

In designer, that means that the output is find in the DOS-box/console.

In the runtime, the logging can be found in the tomcat logs. Using Windows, a file with the name <tomcat>-stderr.<date>.log should exist. Using Linux, the information should be found in the log catalina.out.
