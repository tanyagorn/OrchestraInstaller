Housekeeping


Topic content

To not flood/overcrowd the Orchestra runtime database, the runtime data will be deleted or archived to a special archive database after a certain period of time. This means that the information of processes is not infinitely available in the monitor. If the long time archive is not active, the entries will be deleted from the orchestra runtime tables after a certain time. If LTA is active, the data is moved to a own Orchestra LTA database. More information about the long time archive can be found here: LTA

The housekeeping and the long time archive can be configured in the environment settings. You can find useful explanations here and here.

Configuration groups

In this section, the groups that exist inside the group "Housekeeping" in the environment settings are explained.

SystemEvent

These are the housekeeping settings for system events. The entries for the system events that are displayed in the monitor are depending on the settings from that group.

BusinessTransaction

These are the housekeeping settings for business transactions. Business transactions are transactions that are explicitly created in a process by using startTA and undoTA. Currently this is not used often.

PricessCleanup

These are the housekeeping settings for process events. It is dealt with every types of process but a difference is made between successful and aborted.

MessageStore

These are the housekeeping settings for the message store. The messages are treated independently from the processes in a way. That is the reason why the messages have their own group. For more information about that (there is a difference which version you are using), read the next chapter.

ESB

These are the housekeeping settings for the enterprise service bus. It is only important with cell communication. Orchestra protocols the signal sending in order to prevent duplicates in the signal sending. This will be cleaned here.

LongtimeArchive

These are the housekeeping settings for the long time archive. Like the name tells, this section is for the long time archive. In LTA there is no difference between processes and messages.

Correlation between processes and messages

At runtime, a message belongs to a process. The message should only be deleted after a process is deleted. For that reason, messages stay in the database as long as the corresponding processes. In older orchestra versions, processes and messages were treated completely independent. The parameter MessageStore.cleanup belongs to that time. Look at the following parameters in the environment settings:

<parameter name="ProcessCleanup.SUCCESS.cleanup"     value="1,HOUR"       type="PERIOD"/>
<parameter name="MessageStore.cleanup"                    value="10,MINUTE"   type="PERIOD"/>

 

In newer versions, messages will be kept for  ProcessCleanup.SUCCESS.cleanup + MessageStore.cleanup. The parameter MessageStore.cleanup should not be configured smaller than 10 minutes. It is the best not to change the delivered parameters.
