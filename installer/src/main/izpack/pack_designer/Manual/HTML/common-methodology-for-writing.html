Common methodology for writing best practice documents


Topic content

1.Start with a clear problem statement

oThe introduction of the document shall introduce the reader to a general problem that is to be solved by this best practice guide

oBy reading this statement the reader shall be put in the position to decide if it is worthwhile to continue reading, possibly because the reader is interested to solving this problem also in his own environment

 

2.Define a clear and simple use case that can be used to demonstrate the problem statement

oThe problem statement shall be transformed into a general example use case

oIf possible the use case should be sufficiently general and not specific to a particular business (e.g. health vs industry) if this is not necessary

oA use case that discusses the use of HL7 or a DICOM worklist is possibly so specific that it is not interesting for a reader not located in Health-Industry

oHowever, a use case that discusses the implementation of a Webservice should not choose an Example of health-industry but rather a generic sample like Weather-Forecast or sth similar that does not require specific knowledge

3.Define what you need in order to implement this use case in Orchestra

oWhat type of infrastructure do we need (Database, Webserver, LDAP, etc.)

oWhat specific concepts of Orchestra are used to solve the problem (e.g. message mapping, channel, cluster, cell communication, etc.)

4.Instead of introducing dry concepts the user should gain some real hands on experience with Orchestra by following the best practice document

oExplanation of concepts is important, but explanation itself is not sufficient

oThe real experience comes from using Orchestra and implementing something

5.Define the skill set that a reader should own to implement the described use case

oDepending on the actual use case the requirements for the reader’s skills might be different

oThe guide should not teach general things like using an operating system, a database query or shell console, however if sth like this is required it should be pointed out beforehand

oIf specific knowledge is required this should be listed with some rating: beginner, advanced, expert

6.Previous to showing the actual solution, list all steps that have to executed in order to solve the problem in Orchestra

oBefore going into details, the reader should have an overview of the total number of steps required to implement a use case

oThis can then be used as a checklist

7.Implement the use case in Orchestra by detailing out and describing all steps listed above.

oThe steps that have been defined in the above checklist are now described in detail

oThe same numbers and step title shall be used as headline, e.g.
       Step 1: Import WSDL Definition

 Step 2: Import XSD Schema
 Step 3. Etc.

8.For terms that are specific to Orchestra make use of a central glossary or the reference manual

oConcepts and terms that are not general language should be listed in a glossary

oSpecific Orchestra Objects that are already described in the reference manual have to be linked using a hyper link to this specific documentation
