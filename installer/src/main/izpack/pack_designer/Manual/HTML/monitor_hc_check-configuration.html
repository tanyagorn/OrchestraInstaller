Check Configuration


Topic content

The configuration of the check can be divided in common parameters, which are necessary for every check and type specific  parameters. In following all configurations get explained detailed. Every parameter signed with "*" cannot be empty.

Common Parameters:

•Type*: Either FileCheck or DatabaseCheck. Every type of check refers to another part of system. This parameter also changes the necessary configurations.

•ID(name)*: A unique name for this check.

•Description*: A short description for this check, which explains its intention.

•ThresholdType*: The Threshold type is commonly set automatically. Depending on your configurations the result has a type. Possible types are Boolean, string and number. E.g. the databaseChecks* delivers a number, which means the threshold values also have to be numbers.

•Operator*: Possible Operators are  > (greater), < (less), = (equal), != (not equal)

•Warning Value*: The value of the warning threshold. The type of depends on ThresholdType, it can be a String , a number or a boolean expression(true or false)

•Error Value*: The value of the error threshold. The type of depends on ThresholdType, it can be a String , a number or a boolean expression(true or false)

•Solution*: A solution hint for a appearing problem. This text is just shown if something went wrong(warning or error). It should contain a hint how to solve the problem or what to do as next step.

 

To evaluate a traffic light state two thresholds are necessary. The evaluation works with this schema:

If  <value returned from a command or execution on the system to check><Operator ><Threshold for warning> is a true statement, the check evaluates a warning as result.

If   <value returned from a command or execution on the system to check><Operator ><Threshold for error> is a true statement, the check evaluates a error as result.

 

Examples:

Threshold type number:

Result Value = 5, Warning Threshold =3, Error threshold =6, Operator >(greater):

5 > 3                ->        true statement        ->        warning

5 > 6                ->        false statement        ->        no error

>>The check evaluates a warning as result

 

Result Value = 1000, Warning Value=1500, error Value=1100, Operator <(less):

1000 < 1100        ->        true statement        ->        warning

1000 < 1500        ->        true statement        ->        error

>>The check evaluates an error as result(warning is also true but overwritten by the error)

 

Threshold type string(for this threshold type only operators equal and not equal is allowed):

Result Value = "abc", Warning Value="xyz", error Value="yz", Operator =(equal):

"abc" = "xyz"        ->        false statement        ->        no warning

"abc" < "yz"        ->        false statement        ->        no error

>>The check evaluates a success as result

 

Threshold type boolean(for this threshold type only operators equal and not equal is allowed):

Because boolean can be either true or false three kind of states aren't possible. You have to decide if you want a error or warning message in case of a anomaly.

Result Value = false, Warning Value=false, error Value=true, Operator !=(not equal):

false != false        ->        false statement        ->        no warning

false != true        ->        true statement        ->        error

>>The check evaluates a error as result

 

 

FileCheck

The file check can test your file system. Generally it can be divided in checks, which count files and checks, which ensure that something exists.  

Parameter:

•Execution Type*: This describes the kind of check. Do you want to count files in a path or do you want to check is a path or file exists?

•Path*: The path to check.

•Check parameter*: Depends on execution type:        At execution type count you can add *, .<file_extension>, xxx*xxx

                                                 At execution type exists you can add either nothing to check if the path exists or the name of                                                                the file, which should exist.

•Writeable: If this parameter is set to true the check additionally improves if the path or in case of execution type exists the file are writeable.

DatabaseCheck

 

Database checks execute statements on the runtime database of Orchestra, which is set in environment_settings.xml. The main tables of a running Orchestra runtime  get checked.

To avoid wrong or long running executions the statements aren't changeable. The check executes a count statement to find the number of wrong records in a table. If this number is to big a select statement is executed to find the causing records and show them to you. Following parameters are set:

Parameter

•Category: The category of check. All database table belong to  one part of running Orchestra, e.g. messages, queuing or housekeeping.

•Table:The table to execute the check on.

•Condition: The condition of the statement: select COUNT(*) from <table> where <condition>

•Causer: A second statement to find the causing records in table. This is usually a normal select statement and just executed it the first statement evaluates an error or warning.
